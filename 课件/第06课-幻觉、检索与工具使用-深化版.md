## 上节回顾

Prompt 可以把模型用得更好，但它解决不了一个根本问题：

**模型可能仍然会流畅地说错。**

所以这节课从“怎么提问”切到“怎么让系统更可靠”。

---

## 什么是幻觉

### Hallucination

AI 幻觉指的是：

- 内容表面流畅，形式上像真的
- 但事实、来源或推理过程出了问题

常见类型：

- 事实性幻觉
- 引用/来源幻觉
- 推理性幻觉
- 前后不一致

---

## 为什么会发生

最核心的原因还是第 1 课讲过的那句：

> 模型首先是在做“最像下一个 token”的预测，不是在做“真伪校验”。

再加上几个现实因素：

- 训练数据本身有噪声
- 知识有截止时间
- 上下文不完整
- 用户问题里可能带着错误前提

---

## 更深一层：参数记忆不是可验证记忆

语言模型把知识存进参数后，调用方式是分布式的、统计性的：

- 它不会像数据库那样按主键精确查表
- 也不会天然附带“来源链接”和“证据位置”

所以从系统角度看，参数知识更像：

> 压缩后的模式记忆，而不是可审计、可索引、可验证的外部记忆。

这就是为什么模型可能“知道个大概”，却给不出可靠出处。

---

## 先别急着怪模型

有些错误其实来自任务设置本身：

- 问题要求了它不具备的实时信息
- 需要外部数据库，但系统没接
- 用户给的信息本来就是错的

所以工程上一个更成熟的问题应该是：

**这个任务该不该只靠裸模型完成？**

---

## RAG 的核心直觉

### Retrieval-Augmented Generation

不要只让模型“靠记忆回答”，而是：

1. 先检索相关资料
2. 再把检索结果塞进上下文
3. 让模型基于外部证据生成答案

一句话总结：

> RAG 不是让模型更聪明，而是让模型回答时多一双“翻资料”的手。

---

## RAG 更深一层：把记忆拆成两部分

从系统设计视角，RAG 本质上是在把“知识”拆开：

- 参数里保留通用语言能力和推理模式
- 外部索引里保留可更新、可追溯的事实材料

于是回答过程可以写成两段：

1. 检索：$d_1,\dots,d_k = \text{Retrieve}(q)$
2. 生成：$y \sim p_\theta(y \mid q, d_1,\dots,d_k)$

所以 RAG 解决的不是“模型不会说话”，而是：

> 不要让模型独自承担事实存储与事实调用这两件事。

---

## 一个最小 RAG 流程

```text
用户问题
-> 检索器召回文档
-> 选出最相关片段
-> 把片段放进 prompt
-> 模型生成答案
-> 引用来源 / 返回结果
```

RAG 适合解决的典型问题：

- 企业知识库问答
- 文档问答
- 需要来源可追溯的场景

---

## 检索为什么常常比生成更先成为瓶颈

很多系统里，真正决定上限的不是生成器，而是：

- 有没有召回到对的文档
- 文档切分是否合理
- 重排是否把关键证据放进上下文

可以粗糙地把系统误差拆成：

> 没找到 + 找到了但没放进去 + 放进去了但没用好

所以 RAG 评估不能只盯回答文本，还要盯检索链路本身。

---

## 工具使用和 RAG 的区别

RAG 更像：

- 去资料库里找文本证据

Tool Use / Function Calling 更像：

- 调计算器
- 查数据库
- 调接口
- 执行工作流

两者都在做同一件事：

> 让模型把“不会的事”交给外部系统完成。

---

## Tool Use 更深一层：模型在学“选择动作”

如果讲得更抽象一点，工具调用可以理解成：

- 模型先预测一个动作 $a$
- 再由环境执行 $a$
- 最后把返回结果继续放回上下文

也就是：

$$a \sim p_\theta(a \mid x), \quad o = \text{Env}(a), \quad y \sim p_\theta(y \mid x, a, o)$$

所以工具使用不是“模型自己会查物流”，而是：

> 模型学会了什么时候该把问题路由给外部动作空间。

---

## Function Calling 的直觉

模型不是直接替工具干活，而是先决定：

- 要不要调用工具
- 调哪个工具
- 传什么参数

例如用户问：**帮我查一下订单 1024 的物流状态**

更可靠的系统做法是：

- 模型识别“这需要查系统”
- 触发 `query_order_status(order_id=1024)`
- 再把工具返回结果组织成人类可读回答

---

## 为什么 Tool Use 比纯 Prompt 更像系统工程

Prompt 主要在调模型内部已有分布。

Tool Use 则是把模型嵌进一个闭环系统里：

- 模型负责决策和语言接口
- 外部工具负责真实执行和状态更新

所以这类系统的本质更像：

> LLM + planner/router + external environment

这也是 Agent 系统复杂度陡增的根源。

---

## 可靠性不只靠架构，还靠评测

真实系统上线前，至少要看这些指标：

- 正确率 / 引用命中率
- 是否拒答应该拒答的问题
- 工具调用成功率
- 长文档下是否丢信息

很多问题不是“模型不行”，而是：

- 检索没召回
- 文档切分太差
- 工具参数设计不合理

---

## 人类校验仍然重要

尤其在高风险场景：

- 医疗
- 法律
- 金融
- 合同和审计

合理的默认态度不是“全信 AI”，而是：

> 让 AI 先帮你缩小范围，再由人做最终确认。

---

## 本节小结

> 第6课把“模型会错”升级成“系统怎么减错”。

- 幻觉来自概率生成目标、知识边界和上下文不足
- RAG 通过把参数记忆和外部记忆拆开来缓解“靠记忆乱答”
- Tool Use 通过外部动作空间弥补模型能力边界
- 可靠系统还需要评测、监控和人工校验
